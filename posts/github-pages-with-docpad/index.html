Deploying to Github Pages With Docpad

Thu Sep 26 2013:

Lately, a lot of developers are shedding the weight of database driven sites for simple projects like blogs and static websites. I've really been enjoying Docpad. It's built on top of node and is a really simple and clean way to generate a static website.

Docpad is similar to Octopress or Jekyll in that it allows you to write posts/pages in actual files, then run a build process to compile your site into a single static folder.

Why Static Websites?

Sometimes, you just don't need anything more than that. If you don't require a database, it's much easier and faster to generate the static pages and let the web server do the work.

You get to take advantage of caching, no database connection, security and simplicity.

Not to mention, not ever having to use the Wordpress post editor again. Writing markdown in your text editor! Heyoo...

Docpad's Build Process

Docpad builds a publish folder called 'out'. That folder is a well organized structure of all your compiled assets in a public root level folder.

It takes care of your templating, your partials, compiles your Sass, Jade, Markdown or whatever you decided to use, and builds it into standard HTML, CSS and Javascript.

Deploying with Docpad

Docpad offers a few different deployment options. I like the Github Pages option for a few reasons:

  • Keeps your entire working and public project in Github
  • Fast, reliable and fully managed servers
  • Versioned, forkable, trackable repos
  • Oh yeah, and it's free!

Github Pages is well covered on their site. But there's a few things you need to know before you decide to host there.

User/Org Pages or Project Pages

You can host your site in one of two ways. As a user/org or as a project. Its pretty self-explanatory. If you want your Pages to be general for your identity. (Think, root public folder...) You probably want to use the user/organization.

If your site is related to a specific project instead. Use the Project Pages.

What's The Difference?

For the viewer, the only difference is the URL. Your user/org page URL will be username.github.io If you set up a project page, the URL will be username.github.io/project.

Technically, there's a couple of differences. User/org pages live in a separate repo, while a project page will only be a separate branch in your project repo.

Github will automatically make any Github Pages Repo Public, so if for any reason your project is Private, you probably want that separate repo for your site.

Project pages are automatically recognized by having a branch called gh-pages in your repo.

However, user/org pages live in the master branch of a repo with the naming convention: username.github.io

Once you've decided which one you're going to go with. Its time to get started.

Install the Github Pages Deploy plugin

In your terminal, navigate to your Docpad site folder and install the plugin:

$ docpad install ghpages

You'll get the npm install reporting:

info: The action completed successfully
info: Shutting down... cya next time!

You can check out the source on the Docpad Github Pages Plugin Page. But don't be deterred by the fact that it says it doesn't yet support user/org pages. Because it does. Out of the box.

Setting up a Project Page

If you want to host it as a project page. You're literally done. Just type:

$ docpad deploy-ghpages --env static

And it will read your remote. And push to a new gh-pages branch! Go to username.github.io/project and soak it in.

Setting up a User/Org Page

If you're going to set up a separate repo, with a user/org page. Go to your Github account and create a new repo called username.github.io. (yes, your username, not the actual string)

After you've created the repo. Open your local repo in github and add the new remote location:

$ git remote add pages git@example.com:my_project.git

I like to call that repo pages. Assuming you have your SSH Keys set up correctly, you should be fine for pushing there automatically.

Now you're going to edit your /docpad.coffee config file. If you don't have a plugins section, add it. Then put your ghpages config below it:

plugins:
  ghpages:
    deployBranch: 'master'
    deployRemote: 'pages'

This tells the gh-pages plugin to push to your remote called 'pages' and do it directly to the branch called 'master'.

Give it a shot!

$ docpad deploy-ghpages --env static

It should automatically set up your 'static' environment and push your code to your 'pages' repo.

Don't forget to commit all your changes to your 'working' repo as well though.

$ git push origin master

Conclusion

That's it. You now have a full deployment process to Github Pages on your Docpad site. Let me know if you have any questions...
