Preparing Your Website For Deployment

Fri Sep 20 2013:

One of the practices that's really been improving in the javascript/css/html world is build processes.

If you're on the beginning side of web development, they can be an intimidating thing. But if you focus on understanding what it is first. It becomes a lot easier to implement.

What is a build process

The easiest way to think of a build process is -- running an action on your files to help prepare them for production.

We use lots of shortcuts, and libraries, and tricks in our development, and web servers don't need to see them. And in a lot of cases, won't know what to do with the files anyway.

But developing is so much better with separation, shortcuts and organization, we can't let THAT stop us from using it.

So instead. We configure a helper to take the files, lint, compress, concatenate and copy the files into a pretty and organized folder full of html, css and javascript. Exactly how a web server wants to read its' files.

Structure

Using a build process will allow you to separate your 'working folder', where you do your coding, from your 'production folder' that the web server will read.

Some common naming conventions for the two folders:

  • src/out
  • app/dist
  • working/build
  • pre/public
  • etc...

They can have any names, it's just important for them to be obvious.

Features

Build processes can do a lot of things. And any time its automated, you're decreasing your chance of making a mistake. And giving yourself more time to write code!

Compiling

  • Write your code in SASS, CoffeeScript, Jade, or anything you like.
  • You can easily configure your build script to compile that code into standard html, css and javascript

Compressing

  • Running your code through 'minifiers', will take those easy to read variables, and those beautiful line breaks, and squash them into a single 'ugly' line.
  • Why? Filesize baby. Cutting out those extra letters is less for a server to think about
  • There's no actual benefit for a computer to read them pretty. They just don't care. Better to have it load as quickly as possible instead.

Concatenating

  • Likewise, while its ideal to separate your CSS and Javascript files into smaller pieces so you can use them more easily. Again, your server just doesn't care
  • Concatenating will combine your files into a single file of that type, and only have to load it once.
  • This is way better for performance as limiting http requests is always a good thing.

Copying

  • Move your compressed and concatenated files into a folder structure that makes more sense for URLs instead of what works best for you

Linting

  • These scripts can scan your code for common errors, missed punctuation, spelling etc...
  • Trust me, now matter how good a developer is, they will make mistakes! Linters, always catch them.

Tools

While you can write these build scripts yourself. If you're reading this article, you probably need some practice in using them first. Here's some build script libraries to get you started.

  • GruntJS -- An awesome task manager in javascript
  • Brunch -- An equally awesome collection of build processes
  • Codekit -- An incredible GUI for build processes

And lots of site building frameworks have it built in!

  • Yeoman -- A workflow for building modern apps
  • Docpad -- A framework for building static sites (That's what I use here)
  • Asset Pipeline -- Rails build process for Javascript/CSS
  • Octopress -- A cool static site generator that works with Github Pages

There's a lot more out there. But I've personally used, and can vouch that these ones are great. But definitely explore everything out there!

Conclusion

It might be a hurdle to get into using a build system. But once you do. Trust me. You will never build a site without one again.
